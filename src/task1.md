# General QA Questions
**Question 1:** What are the key responsibilities of a QA Engineer in the software development lifecycle?
**Answer 1:** <!-- <The QA Engineer acts as a quality gate for software ready to be deployed to production. Verifying (did we build the product right) and Validating (did we build the right product) deliverables from the team. Exploring and identifying edge cases, user paths and common scenarios are operating at the very least functionally.>  -->

**Question 2:** What do you think is more important for QA: finding bugs or preventing bugs? Why?
**Answer 2:** <!-- < I do not think one is more important than the other. In my experience they go hand in hand, when a bug is found, I seek to understand the root cause as to why it occured and this helps to create test scripts which check against this moving forward, preventing a reoccurance and improving coverage.> -->

**Question 3:** You find a critical bug on the day of a release. What steps would you take?
**Answer 3:** <!-- <First understand the nature of the bug found, replicate and assess and confirm its impact, severity and priority. Next inform all stakeholders with relevant information, options and directive. It could be the release can proceed with the feature containing the issue disabled whislt developers resolve the bug, alternatively a workaround could be applied so impact is minimal or alternatively the release can be postponed for a few hours for a patch to be tested and applied. > -->

**Question 4:** What would you do if you found a requirement that was unclear or incomplete during testing?
**Answer 4:** <!-- <I would speak to the product owner and development team immediately to clarify the requirment> -->

**Question 5:** How would you manage testing if the development team delivers a feature at the last minute?
**Answer 5:** <!-- <Apply a risk based anaylsis approach to ensure sensitive areas of the product are tested as quickly as possible > -->

**Question 6:** What is exploratory testing, and when is it most useful?
**Answer 6:** <!-- <Exploratory testing is using the application without following a set of test scripts, seeing how the appliation responds intuitively. Very useful when beginners are using an application for the first time, useful when major backend changes have been applied, following a UX/UI redesign etc.> -->

**Question 7:** What is regression testing, and how do you decide what areas to cover during regression testing?
**Answer 7:** <!-- <Regression testing are tests following changes to the application to ensure existing features are still working as expected in addition to newly added features.> -->

**Question 8:** What is a test case, and what details should a good test case include?
**Answer 8:** <!-- <A test case is a set of instructions that a QA engineer uses to carry out a test. It should include the following: an id, title, environment/setup, preconditions, description, expected results, actual results and status> -->

**Question 9:** Have you ever worked with tools for managing test documentation? Which ones, and how did you use them?
**Answer 9:** <!-- <Yes, Excel sheets, Test complete and Jira. A backlog item marked for qa would enter the 'ready for qa' stream once developers had completed the work. Test scripts are written in BDD (Given When Then) in Excel or in Jira and tests manually carried out. Status updated in Excel or in Jira and completed items passed onto the next phase. If scripts where in excel, excel sheets were attached to the respective Jira item. Test complete used to collate testing and provide a summarised report. > -->

**Question 10:** What should be included in a QA test report?
**Answer 10:** <!-- <A QA test report should include a summary of testing carried out for the respective application. Environment/Configruation applied. Total amount of tests carried out, coverage, priority and severity of the tests. Any issues observed, issue priority and severity. Any fixed and retested, summary of application stablity and go or no-go status.> -->

**Question 11:** How do you handle situations where a developer disagrees with a defect you've reported?
**Answer 11:** <!-- <I would have a conversation with the developer to understand their view as it could be by design and not a bug and cross reference against the busisness requirment document> -->

**Question 12:** How do you ensure that stakeholders understand the risk of releasing software with open defects?
**Answer 12:** <!-- <I'd explain the impact of the defects to the business and to the application to the stakeholders and ensured they understood and were okay with the risks involved with the release> -->

**Question 13:** How do you decide the severity and priority of a defect?
**Answer 13:** <!-- <Serverity focuses on the impact on the system. Priority focuses on the impact to the business>-->

**Question 14:** You are tasked with providing a report of testing progress for a release. What would your report include?
**Answer 14:** <!-- <A percentage of test cases completed as part of the release to date. A list of any bugs identified, the severity and priority and status uncovered by testing for the release. Projection on completion of test scripts based on man hours for the release. Projection on retesting raised defects for the release. Any blockers or potential blockers that could hamper the release> -->
